The core idea

A test tree is a test you can read like a sentence

Instead of a flat list of it("does x") calls, contree organises expectations as a tree of when / then branches. It reads top-to-bottom like plain English, but every branch is backed by a real assertion in a real test.

The keywords come from EARS, a lightweight requirements grammar: when an event, if something unwanted, while a state holds, and bare then for always-true rules.

Nesting is causal: a case that can only happen after an earlier outcome sits underneath it. So the duplicate-URL case below lives under the successful-save branch — because that's the only world where it can occur.

One tree maps to exactly one test file, and the file's describe/it structure mirrors the tree word-for-word. The tree and the test never disagree. 

### createBookmark (Use-case)

createBookmark
  when called with a valid URL
       for a signed-in user
    then the URL is canonicalised
    and the bookmark is saved
    and the saved bookmark is returned
    when a bookmark with the same
         canonical URL already exists  # causal nesting
      then the existing one is returned
      and the store is not written to
  if canonicalisation fails
    then a ValidationError is raised
    and the store is never touched

Why it matters

The trees are the requirements — not a doc that rots beside them

Every tree lives in one file at your project root: TEST_TREES.md. That file is the single source of truth for what your system does.

You've probably seen requirements live in a wiki, a ticket, or someone's head — and slowly drift from what the code actually does. contree removes the second copy entirely. There's no separate spec to keep in sync, because the spec and the tests are the same artefact.

After every response, a stop hook asks whether the trees and the code have drifted — and proposes fixes when they have. Your requirements can't quietly go stale, because something is always watching the gap.

Where each test belongs

One standardised architecture, layered outside-in

The question that stumps most people new to TDD is “what kind of test should this be?” contree answers it the same way for every project: a hexagonal structure with five test layers. You start at the outside and let each failing test pull the next layer into being.

Journey test/journey
One curated, max-realism arc across several features — real everything, under 5 minutes. The outside-in entry point for new user-visible work.
System test/system
A single capability, wired as the whole app with real driven adapters. Speaks the user's vocabulary.
Adapter *.adapter.test
One adapter against the real infrastructure it fronts — a database, an HTTP API, the filesystem.
Use-case *.use-case.test
Orchestration logic over fast in-memory stand-ins for its dependencies.
Domain *.domain.test
The pure core — rules and calculations with no I/O at all. Fast and deterministic.

Hexagonal means the domain stays pure, all I/O lives in adapters, and dependencies point inward. Each outbound dependency is a port shipping two versions — a fast in-memory twin and the real adapter — both held to one shared contract, so the stand-in your fast tests use never lies about the real thing. Higher layers never excuse a missing lower one: you implement only once there's a failing test at the behaviour's own layer.

Under the hood

How it actually runs inside Claude Code

contree isn't prompt-engineering you have to remember to apply. It's four hooks wired into Claude Code's lifecycle. Hooks are shell commands the harness runs at fixed events; contree's are plain bash scripts that read the event on stdin and reply over three channels the model can see. Here's the entire wiring — one file: 

// contree/hooks/hooks.json — runs identically under Claude Code and Codex
{
  "hooks": {
    "SessionStart":    [{ "command": "bash $CLAUDE_PLUGIN_ROOT/hooks/session-start.sh" }],
    "Stop":            [{ "command": "bash $CLAUDE_PLUGIN_ROOT/hooks/stop-drift-check.sh" }],
    "UserPromptSubmit":[{ "command": "bash $CLAUDE_PLUGIN_ROOT/hooks/self-care-20-20-20.sh" }],
    "PostToolUse": [{ "matcher": "Edit|Write|MultiEdit",
                      "command": "bash $CLAUDE_PLUGIN_ROOT/hooks/post-update-check.sh" }]
  }
}

Where each hook fires in a Claude Code turn

Claude Code emits named events as it moves through a session. A hook is just a command bound to one event; the harness runs it, hands it the event as JSON on stdin, and reads back its output. Here is one full turn, with contree's hooks marked where they fire:

SessionStart Session opens
Before you type anything. session-start.sh streams the rules and your living docs into the fresh context, so the model starts the session already holding the contract.
UserPromptSubmit You send a prompt
After you hit enter, before the model sees it. self-care-20-20-20.sh records a heartbeat and may prepend an eye-break nudge — otherwise it stays out of the way.
model Claude reasons & calls tools
The model works the request — reading files, running commands, and writing code.
PostToolUse after every Edit / Write / MultiEdit
Fires on each matching tool call. post-update-check.sh looks at which file changed and, if it was MENTAL_MODEL.md, validates it on the spot. (This is the only hook gated by a matcher.)
Stop Claude finishes its reply
When the model is about to hand the turn back to you. stop-drift-check.sh decides whether to let it stop, or to push back a drift check first.
↺ exit 2 Drift check found → back into the model
If the Stop hook exits 2, its message becomes a new turn the model must answer — reconcile the docs, or reply 0. Only once it exits 0 does control return to you. This is the loop you see after each of my responses.

How a hook talks back — stdin in, three channels out

Each script reads the event on stdin and answers over one of three channels. Which channel it uses decides how forcefully the message lands:

stdout · exit 0

Appended to context as background. Quiet — the model reads it but isn't forced to act. SessionStart seeds the session this way.

stderr · exit 2

Fed back as a fresh turn the model must answer before it can stop. Loud — it interrupts. The Stop hook forces its drift check this way.

additionalContext JSON

A {hookSpecificOutput:{additionalContext}} object on stdout, injected only when there's something to say. PostToolUse and UserPromptSubmit use it.

What each script actually does

Every hook is plain bash — no daemon, no state but a few files. Condensed to its essence, here is the real logic of each:

SessionStartsession-start.sh→ stdout · exit 0

Concatenate the operating discipline and the project's living documents, then print them. Claude folds SessionStart stdout into the model's context.

# everything printed here is appended to the new session's context
cat <<'EOF'
  # Directions   — one line per skill: when to reach for change / tdd / sync / setup
  # Rules        — KISS, fail-fast, hexagonal, no-comments, trees-are-the-contract…
EOF

[ -f MENTAL_MODEL.md ] && cat MENTAL_MODEL.md   # the architecture, vocabulary & invariants
[ -f TEST_TREES.md ]   && cat TEST_TREES.md     # the behaviour contract, verbatim

exit 0

Why: the single most expensive failure mode is a model that hasn't read the spec. Injecting it at t=0 means every session — and every parallel agent — starts from the same contract instead of rediscovering it.

Stopstop-drift-check.sh→ stderr · exit 2

Two guards, then the drift check. The guards are what make a re-prompting hook safe to run on every single turn.

INPUT=$(cat)                                   # the Stop event JSON, on stdin

# guard 1 — the harness sets this when a hook's own exit 2 re-triggered the model.
#           bail, or we'd drift-check our own drift-check forever.
jq -e .stop_hook_active <<<"$INPUT" && exit 0

# guard 2 — if my last reply ended in “?”, I'm waiting on the human.
#           yield silently so the question isn't buried under drift output.
last=$(jq -rs 'last assistant text | trim | final char' "$transcript_path")
[ "$last" = "?" ] && exit 0

# otherwise — emit the four reconciliation prompts and force a turn
echo "MENTAL MODEL: did this task reveal knowledge not in the docs?" >&2
echo "TEST TREES:   have trees and implementation drifted?"          >&2
echo "CLAUDE.md:    has it drifted from reality?"                     >&2
echo "README:       is it out of date now?"                          >&2
exit 2                                          # → Claude must answer before stopping

Why: documentation rots because nothing checks it at the moment of change. This makes the check unavoidable and immediate — yet the two guards keep it from looping or talking over a question meant for you. The 0 I reply when nothing drifted is the all-clear.

PostToolUsepost-update-check.sh→ additionalContext

Bound to the Edit|Write|MultiEdit matcher, so it only runs after a file actually changed. It cares about exactly one file.

file=$(jq -r .tool_input.file_path <<<"$(cat)")    # what did the edit touch?

[ "$(basename "$file")" != MENTAL_MODEL.md ] && exit 0  # not the mental model → done

findings=$(validate-mental-model.sh)             # 7 named sections? section caps? rogue headings?
[ -n "$findings" ] && emit_additionalContext "$findings"  # surface only if something's off

exit 0

Why: the mental model is a deliberately constrained artifact — seven sections, capped, no stray headings — so it stays a theory and not a junk drawer. Validating it the instant it's edited keeps that shape without a human reviewer in the loop.

UserPromptSubmitself-care-20-20-20.sh→ additionalContext

No language model, no network — just heartbeat files on disk, so the “how long have you been working” signal is shared across every concurrent session.

touch $HEARTBEAT_DIR/$now              # stamp this moment
rm heartbeats older than 1h           # keep the dir bounded

# the unbroken run of heartbeats with no gap > 5 min = continuous keyboard time
stretch=$(newest_run_without_gap 5min)
[ "$stretch" -lt 20min ] && exit 0     # not at the 20-min mark yet
[ nudged_within 20min ]  && exit 0     # throttle: one nudge per window

emit_additionalContext "look 20ft away for 20s — the 20-20-20 rule"

Why: the contract is about sustainable practice, not just clean code. State lives in files rather than memory, so the timer is honest even when you're running several agents at once.

All four are wired by the one hooks.json above, and every command is prefixed with $CLAUDE_PLUGIN_ROOT — which the harness expands to the install path, so nothing is hard-coded. Codex CLI sets the same variable and fires the same events, which is why one set of scripts drives both harnesses with no changes.

Set expectations

Not a software factory — a harness you steer

contree doesn't turn a prompt into shipped software while you watch. You stay in the loop, working mainly at the level of test trees — the place where intent lives. You start from a vision, not a finished spec, and discover the exact requirements one tree at a time.
