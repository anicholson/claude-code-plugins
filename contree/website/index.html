The core idea

A test tree is a test you can read like a sentence

Instead of a flat list of it("does x") calls, contree organises expectations as a tree of when / then branches. It reads top-to-bottom like plain English, but every branch is backed by a real assertion in a real test.

The keywords come from EARS, a lightweight requirements grammar: when an event, if something unwanted, while a state holds, and bare then for always-true rules.

Nesting is causal: a case that can only happen after an earlier outcome sits underneath it. So the duplicate-URL case below lives under the successful-save branch — because that's the only world where it can occur.

One tree maps to exactly one test file, and the file's describe/it structure mirrors the tree word-for-word. The tree and the test never disagree. 

### createBookmark (Use-case)

createBookmark
  when called with a valid URL
       for a signed-in user
    then the URL is canonicalised
    and the bookmark is saved
    and the saved bookmark is returned
    when a bookmark with the same
         canonical URL already exists  # causal nesting
      then the existing one is returned
      and the store is not written to
  if canonicalisation fails
    then a ValidationError is raised
    and the store is never touched

Why it matters

The trees are the requirements — not a doc that rots beside them

Every tree lives in one file at your project root: TEST_TREES.md. That file is the single source of truth for what your system does.

You've probably seen requirements live in a wiki, a ticket, or someone's head — and slowly drift from what the code actually does. contree removes the second copy entirely. There's no separate spec to keep in sync, because the spec and the tests are the same artefact.

After every response, a stop hook asks whether the trees and the code have drifted — and proposes fixes when they have. Your requirements can't quietly go stale, because something is always watching the gap.

Where each test belongs

One standardised architecture, layered outside-in

The question that stumps most people new to TDD is “what kind of test should this be?” contree answers it the same way for every project: a hexagonal structure with five test layers. You start at the outside and let each failing test pull the next layer into being.

Journey test/journey
One curated, max-realism arc across several features — real everything, under 5 minutes. The outside-in entry point for new user-visible work.
System test/system
A single capability, wired as the whole app with real driven adapters. Speaks the user's vocabulary.
Adapter *.adapter.test
One adapter against the real infrastructure it fronts — a database, an HTTP API, the filesystem.
Use-case *.use-case.test
Orchestration logic over fast in-memory stand-ins for its dependencies.
Domain *.domain.test
The pure core — rules and calculations with no I/O at all. Fast and deterministic.

Hexagonal means the domain stays pure, all I/O lives in adapters, and dependencies point inward. Each outbound dependency is a port shipping two versions — a fast in-memory twin and the real adapter — both held to one shared contract, so the stand-in your fast tests use never lies about the real thing. Higher layers never excuse a missing lower one: you implement only once there's a failing test at the behaviour's own layer.

The day-to-day loop

Five skills carry you from idea to verified code

contree installs as skills your AI assistant reaches for automatically — so the discipline holds even when you're moving fast. Run /contree:setup once, then live in the loop.

Under the hood

How it actually runs inside Claude Code

contree isn't prompt-engineering you have to remember to apply. It's four hooks wired into Claude Code's lifecycle. Hooks are shell commands the harness runs at fixed events; contree's are plain bash scripts that read the event on stdin and reply over three channels the model can see. Here's the entire wiring — one file: 

// contree/hooks/hooks.json — runs identically under Claude Code and Codex
{
  "hooks": {
    "SessionStart":    [{ "command": "bash $CLAUDE_PLUGIN_ROOT/hooks/session-start.sh" }],
    "Stop":            [{ "command": "bash $CLAUDE_PLUGIN_ROOT/hooks/stop-drift-check.sh" }],
    "UserPromptSubmit":[{ "command": "bash $CLAUDE_PLUGIN_ROOT/hooks/self-care-20-20-20.sh" }],
    "PostToolUse": [{ "matcher": "Edit|Write|MultiEdit",
                      "command": "bash $CLAUDE_PLUGIN_ROOT/hooks/post-update-check.sh" }]
  }
}

A hook replies to the harness over one of three channels, and which one decides how forcefully its words land. You'll meet each at the stage that uses it — here's the legend to refer back to:

stdout · exit 0 quiet — appended to context as background
additionalContext structured — injected only when there's something to say
stderr · exit 2 loud — fed back as a turn the model must answer

One session, stage by stage

Claude Code emits a named event at each point in a turn. Walk the turn top to bottom: at every event one contree hook fires — see what it runs, why, and what it changes — then move to the next.

1
The session opens
SessionStartsession-start.shstdout · exit 0

Before you've typed anything, the harness fires SessionStart. The script concatenates the operating discipline and the project's living documents and prints them; Claude folds SessionStart's stdout into the model's context.

cat <<'EOF'
  # Directions   — one line per skill: when to reach for change / tdd / sync / setup
  # Rules        — KISS, fail-fast, hexagonal, no-comments, trees-are-the-contract…
EOF

[ -f MENTAL_MODEL.md ] && cat MENTAL_MODEL.md   # architecture, vocabulary & invariants
[ -f TEST_TREES.md ]   && cat TEST_TREES.md     # the behaviour contract, verbatim

exit 0                                # stdout → appended to the new session's context

Why The most expensive failure mode is a model that never read the spec. Seeding it at t=0 removes that risk before the first prompt.

Impact Every session — and every parallel agent — begins holding the same contract: the architecture, the trees, and when to use each skill. No rediscovery.

The exact text injected into the agent — Directions + Rules, verbatim 
# Directions

Eagerly use these skills to fulfil operator requests, where applicable:

- change — any behaviour change, before any code is discussed or written
- tdd — implementing behaviour, writing code, or writing tests
- sync — drift, gaps, staleness, or completeness
- setup — no test framework is configured or TEST_TREES.md is absent
- workflow — the full arc from idea to verified working software

# Rules

- KISS — complexity is bad; simplicity above almost all else
- YAGNI — don't future-proof; implement only what you need now
- Subtract, don't add — can this be achieved by simplification instead?
- No fake code — no skeletons, placeholders, or temporary implementations
- Avoid indirection — direct is better than conforming to arbitrary patterns
- Data over control flow — push variability into data so the hot path stays uniform. Replace "inspect state, branch, decide" with "do the thing to whatever's there." Conditionals that switch on history are a smell; the steady-state loop should do the same thing every tick.
- Fail fast — don't swallow errors; let the system fail when unexpected things happen
- Retry at the source — the layer closest to the failure retries; every layer above derives its timeout from that layer's worst case and does not retry the same failure class
- Avoid nullability — make things required; don't program defensively
- Explicit and expressive — name for what things do, not how they're implemented
- No comments — descriptive tests and expressive code obviate comments; comments bias agents against change, but trees and tests make the contract explicit so code can change radically.
- Composition over inheritance — no `extends`; use hooks, functional utilities, component composition
- Typing — type everything; no `any`
- Z-index — avoid z-index; good layout doesn't rely on it
- Read docs — use Context7 before using any library; don't guess API usage
- Consumer-driven — implement only what a consumer already needs
- Resolve uncertainty — look directly and remove optionality; don't hedge with fallbacks
- Act, don't ask — if your reasoning already includes the right answer, act on it rather than externalising the decision
- pnpm — use pnpm, not npm, for JS/TS
- Trees are the contract — every expected behaviour and side effect goes in `## Test Trees`; every tree is verified by a test; every test drives the real implementation. If it's observable anywhere (filesystem, network, logs, next invocation), it belongs in the tree.
- Behaviour, not internals — every tree describes what crosses its level's interface (inputs, outputs, side-effects). Never the implementation inside.
- Debugging means a test gap — find the tree path that should have caught the bug, write the failing test, then fix the code.
- Hexagonal — domain pure; I/O in adapters; dependencies point inward; each driven port ships with an in-memory twin
- Functional first — the outermost layer is real, max-validity functional testing; the floor is the Journey — one curated arc, under 5 minutes, spanning capabilities and representative error paths, eventually succeeding.
- Outside-in — start each capability from its outermost failing test; it pulls the inner layers into being, each TDD'd at its native layer.
- Layer completeness — every layer owns complete coverage of its own seam. Journey/functional coverage is never coverage of the layers beneath; implement only once a failing test exists at the behaviour's own ground layer.
- Test layers — Journey (curated cross-capability arc), System (one capability, whole-app, real driven adapters), Adapter (driving mocks app, driven hits real infra), Use-case (in-memory adapters), Domain (pure)
- Shared port contract — one `*.contract.ts` suite per port, imported by both in-memory- and real-adapter test files
- One tree, one test file — each tree reifies exactly one test file; the file's describe/it hierarchy mirrors the tree verbatim
…then the full text of your MENTAL_MODEL.md and TEST_TREES.md is appended below this — the agent starts the session holding all three.
2
You send a prompt
UserPromptSubmitself-care-20-20-20.shadditionalContext

The moment you hit enter, before the model sees your message, UserPromptSubmit fires. No model, no network — just heartbeat files on disk, so the “how long have you been at it” signal is shared across every concurrent session.

touch $HEARTBEAT_DIR/$now              # stamp this moment
rm heartbeats older than 1h           # keep the dir bounded

stretch=$(newest_run_without_gap 5min) # unbroken keyboard time
[ "$stretch" -lt 20min ] && exit 0     # not at the 20-min mark yet → stay silent
[ nudged_within 20min ]  && exit 0     # throttle: one nudge per window

emit_additionalContext "look 20ft away for 20s — the 20-20-20 rule"

Why The discipline is about sustainable practice, not just clean code. State lives in files rather than memory, so the timer stays honest even when you're driving several agents at once.

Impact Almost always a no-op. Once every 20 minutes of continuous work, Claude opens its reply with a brief eye-break nudge, then carries on with your request.

The exact text injected into the agent — the nudge, when it fires 
Before addressing the user's request, mention the 20-20-20 rule: they have
been at the keyboard for a while. Say something like: "Quick eye break: look
at something 20 feet away for 20 seconds (20-20-20 rule)." Keep it brief,
then continue with their request.
delivered as additionalContext, so the model weaves it in rather than being interrupted.
3
Claude works — and edits a file
PostToolUsepost-update-check.shadditionalContext

As the model reasons and calls tools, PostToolUse fires after each one — but this hook is bound to the Edit|Write|MultiEdit matcher, so it only runs when a file actually changed. It guards exactly one file.

file=$(jq -r .tool_input.file_path <<<"$(cat)")    # what did the edit touch?

[ "$(basename "$file")" != MENTAL_MODEL.md ] && exit 0  # not the mental model → done

findings=$(validate-mental-model.sh)             # 7 named sections? caps? rogue headings?
[ -n "$findings" ] && emit_additionalContext "$findings"  # surface only if something's off

exit 0

Why The mental model is a deliberately constrained artifact — seven sections, capped, no stray headings — so it stays a theory, not a junk drawer.

Impact The instant MENTAL_MODEL.md is edited, any structural breakage is fed straight back to the model to fix — no human reviewer required.

The exact text injected into the agent — example findings 
MENTAL_MODEL.md validator findings:
- Missing required section: "Temporal View"
- Section "Invariants" exceeds its cap (12 bullets; max 10)
- Rogue heading not in the seven-section set: "Open Questions"
injected only when the validator finds something; a clean file injects nothing.
4
Claude finishes its reply
Stopstop-drift-check.shstderr · exit 2

As the model goes to hand the turn back to you, Stop fires. Two guards run first — they're what make a hook that re-prompts on every turn safe — and only then does the drift check.

INPUT=$(cat)                                   # the Stop event JSON, on stdin

# guard 1 — set when a hook's own exit 2 re-triggered the model.
#           bail, or we'd drift-check our own drift-check forever.
jq -e .stop_hook_active <<<"$INPUT" && exit 0

# guard 2 — if my last reply ended in “?”, I'm waiting on the human.
#           yield silently so the question isn't buried under drift output.
last=$(jq -rs 'last assistant text | trim | final char' "$transcript_path")
[ "$last" = "?" ] && exit 0

# otherwise — push back the four reconciliation prompts and force a turn
echo "MENTAL MODEL: did this task reveal knowledge not in the docs?" >&2
echo "TEST TREES:   have trees and implementation drifted?"          >&2
echo "CLAUDE.md:    has it drifted from reality?"                     >&2
echo "README:       is it out of date now?"                          >&2
exit 2                                          # → Claude must answer before stopping

Why Docs rot because nothing checks them at the moment of change. This makes the check immediate and unavoidable — while the two guards stop it looping or talking over a question meant for you.

Impact After every substantive response, the trees, mental model, CLAUDE.md, and README are reconciled against what just changed before the model is allowed to rest.

The exact text pushed back to the agent — the four drift prompts, verbatim 
MENTAL MODEL: Did this task reveal any knowledge NOT already described in documentation, tests, and code? Default is no change. If a change is warranted: name which of the seven sections it belongs to (Core Domain Identity, World-to-Code Mapping, Ubiquitous Language, Bounded Contexts, Invariants, Decision Rationale, Temporal View); if none fits, it is not part of the mental model; prefer tightening an existing line over adding a new one; state what is true, not what to avoid; when the target section is at its cap, displace or merge an existing item rather than appending.
TEST TREES: Have test trees and implementation drifted apart? If so, propose solutions.
CLAUDE.md: Has CLAUDE.md content drifted from reality? If so, update it.
README: Is the README out of date now? It should tell consumers what the project is, how to install it, how to configure it, and how to use it. If anything is stale or wrong, update it.
If nothing needs attention, reply 0.
written to stderr with exit 2 — the model must address each, or reply 0.
↺ The loop you've been watching. That exit 2 turns the drift prompts into a new turn the model must answer — fix the docs, or reply 0 for all-clear. Only when the hook exits 0 does control return to you, and the turn completes.

Every command in that hooks.json is prefixed with $CLAUDE_PLUGIN_ROOT, which the harness expands to the install path — so nothing is hard-coded. Codex CLI sets the same variable and fires the same events, which is why one set of scripts drives both harnesses with no changes.

Set expectations

Not a software factory — a harness you steer

contree doesn't turn a prompt into shipped software while you watch. You stay in the loop, working mainly at the level of test trees — the place where intent lives. You start from a vision, not a finished spec, and discover the exact requirements one tree at a time.
