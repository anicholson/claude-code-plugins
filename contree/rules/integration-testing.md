- Journey tests are the outermost, max-realism functional layer and the outside-in entry point: the whole app wired with real driving and driven adapters across a multi-capability user arc that establishes the contexts it touches, passes representative error paths (not every one), and eventually succeeds. Name them `*.journey.test.*` under `test/journey/`.
- System tests are functional testing of a single capability, interior to the journey: real driven adapters at the highest tolerable realism — real infrastructure, real boundaries. Name them `*.system.test.*` under `test/system/`. When breadth at max realism is unaffordable, lean on the journey and push combinatorial detail to inner layers — never to broad in-memory wiring at the System layer.
- Journey and functional coverage is never coverage of the layers beneath: implement only once a failing test exists at the behaviour's own ground layer, sitting under the journey/functional failure that motivated it.
- Adapter tests at the Adapter layer exercise one adapter against its port contract. For driven adapters, that means real infrastructure — Testcontainers, a local service, or recorded cassettes. Not mocks.
- Driven adapters ship in two flavours that share one port contract suite: an in-memory adapter used by Use-case tests, and a real adapter used in production and at the System layer. Both adapter test files import and run the same `*.contract.ts` suite. The real adapter's test file layers on adapter-specific behaviour (timeouts, retries, constraint handling) beyond the shared suite.
- Name adapter tests `*.adapter.test.*` and colocate with the adapter
- *Do not* use environment variables to vary behaviour between test and runtime
